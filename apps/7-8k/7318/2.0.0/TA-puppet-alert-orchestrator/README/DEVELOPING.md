# Puppet Alert Orchestrator Development Guide

This add-on is built with the [Splunk UCC Framework](https://splunk.github.io/addonfactory-ucc-generator/) (`splunk-add-on-ucc-framework`). The UCC framework generates the configuration page, alert action HTML forms, REST handlers, and conf spec files from `globalConfig.json` at build time.

## Setup

Create and activate a virtual environment, then install all development and runtime dependencies:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
pip install -r requirements.txt --target TA-puppet-alert-orchestrator/lib
```

`requirements-dev.txt` installs the UCC framework (`ucc-gen` CLI), `slim` (Splunk Packaging Toolkit) for local packaging, and `splunk-appinspect` for local validation. `requirements.txt` installs the runtime Python libraries into `lib/`, which is where Splunk looks for them at runtime.

## Build

Run the UCC build from the repo root:

```bash
ucc-gen build --source TA-puppet-alert-orchestrator --overwrite
```

This generates the complete, deployable add-on into `output/TA-puppet-alert-orchestrator/`. The `output/` directory is not committed — it is produced fresh on every build and in CI.

**Do not edit files inside `output/` directly.** Any changes made there will be overwritten on the next build. Make changes in the source (`TA-puppet-alert-orchestrator/`) instead.

## Package

To produce a `.tar.gz` for local testing or manual upload to Splunkbase:

```bash
slim package output/TA-puppet-alert-orchestrator
```

This is the same command run by the release CI workflow when a version tag is pushed.

## Alert Action Pattern

Each alert action consists of three files:

| File | Purpose |
|---|---|
| `bin/puppet_run_task.py` | Entry point — generated by UCC, do not edit |
| `bin/ta_puppet_alert_actions/modalert_puppet_run_task_helper.py` | `process_event()` — reads params, resolves settings, calls the action |
| `bin/ta_puppet_alert_actions/puppet_task_action.py` | PE API call logic |

The helper (`modalert_*_helper.py`) is where most business logic lives. It uses two utility functions defined at the top of each helper:

- `override(setting_name, helper)` — returns the alert-level param if set, falls back to the global add-on setting
- `notnone(default, possible_none, helper)` — returns `possible_none` if it has a value, otherwise `default`

## Alert Action Forms

Alert action form fields are defined in the `alerts` section of `globalConfig.json` at the repo root. UCC generates the HTML form files during `ucc-gen build` — do not create or edit `default/data/ui/alerts/*.html` files manually. If those files exist in source, they will overwrite the UCC-generated ones.

Default parameter values (e.g. `param.puppet_environment = production`) are set in `default/alert_actions.conf`, not in `globalConfig.json`. The `defaultValue` in `globalConfig.json` is metadata only and does not affect the Splunk runtime.

## The `pie/` Library

`bin/ta_puppet_alert_actions/pie/` is the internal Python library for communicating with Puppet Enterprise APIs:

| Module | Purpose |
|---|---|
| `pie/orch.py` | Puppet Orchestrator API (tasks, plans, job results) |
| `pie/pdb.py` | PuppetDB API (report data) |
| `pie/rbac.py` | PE RBAC API (token generation) |
| `pie/hec.py` | Splunk HEC (posting action events) |
| `pie/util.py` | URL/endpoint helpers |

All `pie/` functions accept an `ssl_verify` parameter which maps to the **Verify SSL Certificate** add-on setting.

## Adding a New Alert Action

1. Define the new action in `globalConfig.json` under the `alerts` key — fields, labels, types, and help text
2. Add the action stanza to `default/alert_actions.conf` with any parameter defaults
3. Add the action to `readme/alert_actions.conf.spec`
4. Create `bin/ta_puppet_alert_actions/modalert_<action_name>_helper.py` with a `process_event(helper, *args, **kwargs)` function
5. Create `bin/ta_puppet_alert_actions/<action_name>_action.py` with the PE API call logic
6. Run `ucc-gen build --source TA-puppet-alert-orchestrator --overwrite` — UCC generates the entry point `bin/<action_name>.py` and the HTML form automatically
