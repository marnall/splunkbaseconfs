## Description
The Machine Learning Cloud Service Add-on for Enterprise Security provides out of the box machine learning models capable of detecting malicious threats using cloud updated models. It complements Splunk Enterprise Security by creating notable events associated with ransomware. The app leverages cloud services to provide monthly updates to the `base` models shipped with the app to improve detection and expand use cases. A `local` machine learning model can be enhanced by using customer labels and threat intelligence. This is achieved by leveraging transfer learning on the `local` model and intelligent workflow orchestrated by the `Z^2 daemon`.

### Special Considerations

#### Disabling Cloud Service
For air-gapped networks or Splunk Enterprise instances without internet connectivity, it is suggested to disable the cloud update modular input. This typically runs every 7 days and will pull a new version of the base model from the cloud service. To prevent this behavior you can disable the `modelchecker.sh` script in the modular inputs menu to prevent https traffic in these environments.

#### Noisy DNS Datamodels
In some instances it may be necessary to make changes to the `Network_Resolution` datamodel or create a new DNS datamodel to tune out noisy queries like internal server queries and reverse ipv4 and ipv6 name resolution ex: `4.4.8.8.in-addr.arpa` to improve the performance of the correlation search.

These queries increase the size of the datamodel and impact the total time required for a search to complete before the data is passed to the machine learning models. They also increase the storage size of the datamodel and impact memory usage required for the completion of a search job.

These reverse queries are excluded from the notable events and `ml_intel` threat collection by using the `cdn()` macro in the correlation search. This macro can also be used to filter out these domains in the datamodel if desired.


#### MLTK Limits
The default setting of `mlspl.conf` limits the ram  utilization (`max_memory_usage_mb`) to 1024MB, this has been increased for this app with a custom mlspl.conf to increase this to 8192MB to suit the needs of most customers. If your are noticing limits due to the amount of data being passed to the model you have three options: 1) filter out noisy domains by using a custom datamodel, 2) increase the frequency of the search and decrease the window (default: hourly looking back -1h@h) to decrease the size of the search results, or 3) increase the `max_memory_usage_mb` in `mlspl.conf` to a higher value eg: 16384MB to be able to fit all of the data in memory.


### Enterprise Security
The correlation search `Z^2: DGA Query Detection Hourly - Rule` runs hourly to check for dga type A queries in the `Network Resolution.DNS` datamodel, creating notable events for each query identified by src. These dga queries are also saved into the ml_intel collection automatically for transfer learning and improvement of the localized machine learning model `local_dga_rba_custom`. The correlation search `Z^2: Botnet Detection Hourly - Rule` runs hourly to check for botnet connections in the `All_Traffic` datamodel, creating notable events for each connection identified by src. These src ips are also saved into the bot_intel collection automatically for transfer learning and improvement of the localized machine learning model `local_botnet_rba_custom`.

### Review Threat Intel
The DGA Threat Intelligence Review dashboard allows an analyst to quickly re-classify domains that have been classified as DGA for  transfer learning of the local model and is scheduled on an hourly basis. Future enhancements will make use of the fields to migrate unique domains into the signature based alerting of Enterprise Security's Threat Artifacts and CDN detection to quickly label false positives.

## SA-Revelation Custom Algorithm Usage

### RBAClassifier
The RBAClassifier algorithm uses the scikit-learn Multi-layer Perceptron estimator for classification. RBAClassifier uses a feedforward artificial neural network model that trains using back-propagation. This algorithm supports incremental fit.

For descriptions of the `batch_size` , `random_state` and `max_iter parameters`, see the scikit-learn documentation at http://scikit-learn.org/stable/modules/generated/sklearn.neural_network.MLPClassifier.html

Using the RBAClassifier algorithm requires running version 1.3 or above of the Python for Scientific Computing add-on.

#### Parameters

- The `partial_fit` parameter controls whether an existing model should be incrementally updated on not. This allows you to update an existing model using only new data without having to retrain it on the full training data set.
- The `partial_fit` parameter default is False.
- The `hidden_layer_sizes` parameter format (int) varies based on the number of hidden layers in the data.
- The `probabilities` parameter specifies whether probabilities for each possible field value should be returned alongside the predicted value.
- The default value of the `probabilities` parameter is False.

#### Syntax

```
fit RBAClassifier <field_to_predict> from <explanatory_fields> [into <model name>]
[batch_size=<int>] [max_iter=<int>] [random_state=<int>] [hidden_layer_sizes=<int>-<int>-<int>]
[probabilities=<true|false>] [activation=<str>] [solver=<str>] [learning_rate=<str>]
[tol=<float>] [momentum=<float>]
```
You can save RBAClassifier models by using the `into` keyword and apply it to new data later by using the `apply` command.

You can inspect models learned by RBAClassifier with the `summary` command.

`... | summary My_Example_Model`

You can view the predicted probabilities of the possible classes from the `apply` step by including the parameter.

`... | apply My_Example_Model probabilities=true`

#### Syntax constraints

- If `My_Example_Model` does not exist, the model is saved to it.
- If `My_Example_Model` exists and was trained using RBAClassifier, the command updates the existing model with the new input.
- If `My_Example_Model` exists but was not trained using RBAClassifier, an error message displays.

#### Example

The following example uses RBAClassifier on a test set.

`... | inputlookup diabetes.csv | fit RBAClassifier response from * into RBA_example_model hidden_layer_sizes='100-100-80' |...`

The following example includes the `partial_fit` command.

`| inputlookup iris.csv | fit RBAClassifier species from * partial_fit=true into My_Example_Model`

### Connection
The Connection algorithm uses the ML-SPL API to create a connection to the Security Machine Learning Cloud Service. If the connection is successful it returns `connection_status` with a value of `true`.

Using the Connection algorithm requires running version 2.0 or above of the Python for Scientific Computing add-on.

#### Parameters

- No parameters are currently supported for the Connection algorithm

#### Syntax

```
fit Connection
```

You cannot save Connection models using the `into` keyword.


#### Syntax constraints

- If connection to the cloud service is unavailable: `connection_status` returns `false`.

#### Example

The following example uses Connection on a test set.

`... | fit Connection `

### Localize
The Localize algorithm uses the ML-SPL API to create a local copy of the base model that can be used for transfer learning of newly labeled ransomware and cloud domains for detection and suppression of false positives. If the duplication is successful it returns `copy_status` with a value of `true`.

Using the Localize algorithm requires running version 2.0 or above of the Python for Scientific Computing add-on.

#### Parameters

- The `base` parameter is the model you are using as the source of information, the default is `base_dga_rba_custom`.
- The `local` parameter is the model you are creating for localized transfer learning, the default is `local_dga_rba_custom`.

#### Syntax

```
fit Localize base=<str> local=<str>
```

You cannot save Localize models using the `into` keyword.


#### Syntax constraints

- If duplication of the model fails: `copy_status` returns with the exception for debugging.

#### Example

The following example uses Localize on a test set.

`... | fit Localize base=my_base_model local=my_local_model`

## SA-Revelation Content Development

The correlation search `Z^2: DGA Query Detection Hourly - Rule` used to create notable events has several different pieces  to intelligently alert users to newly detected domains, while minimizing false positives and ensuring customers can easily integrate new information for retraining of the `local` model.

SPL Query:
```
| tstats summariesonly=true sum(DNS.query_count) as "count" from datamodel="Network_Resolution"."DNS" by "DNS.src","DNS.query",_time
| rename "DNS.src" as "src","DNS.query" as "domain"
| `ut_shannon(domain)`
| `ut_meaning(domain)`
| `ut_domain_parse(domain)`
| apply app:base_dga_ngram
| apply app:base_dga_pca
| `apply_base_local`
| `dga_confidence_filter(0.90,0.80,0.70,0.90)`
| `ml_intel_check`
| eval description="{class:dga, xref:SA-Revelation, dga_probabilty:"+dga_confidence+"}", threat_key="dga_domains"
| table _time src domain probable_class threat_key description
| rename probable_class as class
| outputlookup ml_intel append=true
```

URL Toolbox has several macros that are used for string extraction and scoring different features for a domain. `ut_shannon` provides a shannon entrophy score for the string, `ut_meaning` leverages a dictionary to score the string against a list of words in the english language to determine a ratio of all the words that make up the domain. More information on URL toolbox can be found on [splunkbase](https://splunkbase.splunk.com/app/2734/#/details).

`ut_domain_parse(domain)` is a macro that ships with SA-Revelation and creates several different features based on the string: `ut_vowel_ratio`, `ut_domain_length`, `ut_digit_ratio`, `ut_vowel_ratio`, `ut_consonant_ratio`. Each of these newly created fields are measurements of the string that is passed to the macro. In this case we are measuring different characteristics of domains to classify them as `legit` or `dga`.

`app:base_dga_ngram` uses a TFIDF model to parse bi-grams in the domain, this information is passed to `app:base_dga_pca` to reduce the dimensionality from 300 columns to 3. The resulting columns `PC_1`,`PC_2`,`PC_3` are used as feature inputs for classification of the domain.

`apply_base_local(class)` is a macro that is made up of two models and a custom weighting formula used to determine the likelihood a domain is `legit` or `class`.

`apply_base_local` Macro Contents:
```
apply app:base_$class$_rba_custom probabilities=true
| rename "predicted(class)" as b_class, "probability(class=$class$)" as base_$class$_p, "probability(class=legit)" as base_legit_p
| apply app:local_$class$_rba_custom probabilities=true
| rename "predicted(class)" as l_class, "probability(class=$class$)" as local_$class$_p, "probability(class=legit)" as local_legit_p
| table _time src class b_class base_$class$_p base_legit_p l_class local_$class$_p local_legit_p *
| eval model_divergence_$class$ = ((base_$class$_p - local_$class$_p)/2), model_divergence_legit = ((base_legit_p - local_legit_p)/2)
| eval model_divergence = model_divergence_$class$ + model_divergence_legit
| eval d_class = case(model_divergence>= 0.05, b_class, model_divergence<=-0.05, l_class, 1=1, l_class)
| eval $class$_probability = ((base_$class$_p + local_$class$_p)/2), legit_probability = ((base_legit_p + local_legit_p)/2)
| eval probable_class = case( $class$_probability>legit_probability, "$class$", $class$_probability<legit_probability, "legit", 1=1, d_class)
```
#### DGA Example Usage
The `apply_base_local(dga)` macro is a combination of two Multi-layer Perceptron estimators that have been customized to provide the class probabilities when inferencing. Specifying the parameter `probabilities=true` will output the probability for each class. For example the domain `github.com` has a `probability(class=dga)` of `0.0566851254863735` and a `probability(class=legit)` of `0.9433148745136265`. These two probabilities are renamed as `base_dga_p` and `base_legit_p` as they come from the `base_dga_rba_custom` model, the inference is renamed to `b_class`.

The same pattern is repeated with the `local_dga_rba_custom` model with the probabilities renamed `local_dga_p`, `local_legit_p` and the inference as `l_class`.

#### Botnet Example Usage
The `apply_base_local(botnet)` macro is a combination of two Multi-layer Perceptron estimators that have been customized to provide the class probabilities when inferencing. Specifying the parameter `probabilities=true` will output the probability for each class. For example the src `198.190.116.142` has a `probability(class=botnet)` of `0.0566851254863735` and a `probability(class=legit)` of `0.9433148745136265`. These two probabilities are renamed as `base_botnet_p` and `base_legit_p` as they come from the `base_botnet_rba_custom` model, the inference is renamed to `b_class`.

The same pattern is repeated with the `local_botnet_rba_custom` model with the probabilities renamed `local_botnet_p`, `local_legit_p` and the inference as `l_class`.

The macro algorithm weights the `base` and `local` models' class probabilities using the following formulas to calculate the `probable_class` by leveraging at the confidence of the inference.

![alt text](screenshots/SA-Revelation_probability_formula.png "screenshot of formulas")

 `model_divergence`is calculated to determine which class to leverage as the base model and the local model will diverge from one another as subsequent re-training changes the make up of the neural net. This is done for two reasons: 1. `low_confidence` inferences are likely to return conflicting base inferences: `b_class` and local inferences `l_class`, 2. `base` and `local` models that have a large delta in the number of observations that have been used for subsequent re-trainings are likely to conflict on the probable_class.

To solve for this we can deconflict the models using the formula included below to determine the delta between the models and return the model with the higher confidence, we can also filter low confidence inferences using the `confidence_filter()` thereby reducing the possibility of this occurring frequently.

![alt text](screenshots/SA-Revelation_divergence_formula.png "screenshot of formulas")

`confidence_filter` Macro Contents:
```
eval $class$_confidence= case($class$_probability>=$high$, "high_confidence", $class$_probability>=$medium$, "medium_confidence", $class$_probability>=$low$, "low_confidence", 1=1, "no_confidence") 
| where $class$_probability>=$p$
```

The `confidence_filter(5)` has five arguments total, one string argument and four arguments that are all numerical values from 0 to 1:
  - $class$, string value of the class type eg: dga, botnet used to crate `dga_probability`, `botnet_probability`
  - $high$, minimum value of `$class$_probability` for the inference to be considered `high_confidence`
  - $medium$, minimum value of `$class$_probability` for the inference to be considered `medium_confidence`
  - $low$, minimum value of `$class$_probability` for the inference to be considered `low_confidence`
  - $p$, minimum value of `$class$_probability` to generate notable events

As an example the setting found in the botnet coorelation search: `confidence_filter(dga,0.90,0.80,0.70,0.90)` would mean a domain would have to score above `0.70` for low, `0.80` for medium,` 0.90` for high. The correlation search would only create notable events in Enterprise Security for domains that score above `0.90`.

As an example the setting found in the botnet coorelation search: `confidence_filter(botnet,0.90,0.80,0.70,0.90)` would mean a src would have to score above `0.70` for low, `0.80` for medium,` 0.90` for high. The correlation search would only create notable events in Enterprise Security for ips that score above `0.90`.

The `ml_intel` lookup is used to check for false positives and serves as a suppression list to reduce noisy domains that have already been labeled as `legit` in the Threat Intel Review dashboard by creating a field called `label_class`. Unseen or previously unlabeled dga domains are assumed to be `dga`.

Domains are checked one final time against a known list of cloud providers and internal server naming conventions using the `cdn_list`. Domains that make it past this check will be candidates to populate into the notable events for analyst review.

`probable_class` is derived from the divergence and class probability algorithms above. It is renamed to `class` before the new notable alerts are created. These `probable dga domains` are saved into the `ml_intel` threat collection for analyst labeling and investigation as inferenced threat intelligence.
