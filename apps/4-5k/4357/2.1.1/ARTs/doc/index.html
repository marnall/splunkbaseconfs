What is ARTs?

ARTs (Automation Result Triage system) is a Splunk application for automation test results analysis with the following main features:

  • Latest test results stats
  • Test results stats trend
  • Failure triage
  • Different branch result comparison
  • Global message broadcast

Getting Started

This guide assumes that you know the basic knowledge of Splunk, e.g. how to install and configure Splunk (standalone or clustering) and how to install the Splunk app.

Prerequisites

Assuming that you have installed and configured the Splunk(cluster) environment. Before you are going to install the app of ARTs, you need to complete the following two steps first.

  1. Install the Splunk App for Jenkins
  2. Install the Status Matrix Custom Viz

Configure ARTs

After you install the app of ARTs, you can enable the JIRA settings in the setup page for the very first time. Or you can update the settings in the Manage Apps page.

Note: It is optional. If you don’t enable it, ARTs could not be get information from JIRA system.

If you are not a admin user, you SHOULD assign the role list_storage_passwords capability to user, otherwise it will use the default user/password in jirabugstatus command to access JIRA system. It may fail due to authencication failed or it will affect the performace.

Besides that, you can make some personalized customization based on your requirements. Here are some configuration files.

After updating the files, you might need to restart Splunk Enterprise for the changes to take effect.

arts_settings.conf

# This file contains attribute/value pairs of general personalized customization for ARTs  
[triage]  
# enable/disable html report link/button  
enable_html_report=0  
  
# enable/disable pytest script button  
enable_pytest_script=0  
  
# admin user for ARTs,which have delete permission in global message  
admin_user=  
  
# blacklist for JIRA ticket (splitted with |), the tickets with the following  
# prefix will not be regarded as product bug, such as TNT-123, INFRA-123  
jira_bug_blacklist=TNT|INFRA

macros.conf

In the “Customization Section”, you can update the macros according to your own metadata.

# Update the following macros according to your own metadata if needed  
# --------------------------- Customization Section Start----------------------
...
# In this macro, we extract the error message from the pytest stacktrace
...
# Update the following macros to define the failure category

# This macro will filter out the jobs without test results
...
# --------------------------- Customization Section End----------------

failure_category.csv

The lookup file contains the known infrastructure (connection/start/restart) issues.
For example:

line1 category,type,filter  
line2 connection,errorinfo,SSLHandshakeError  
line3 connection,errorinfo_message, Failed to install/upgrade from
...
line9 start,errorinfo,CouldNotStart

line1: the field names of the lookup file
line2: means the errorinfo contains SSLHandshakeError will be regarded as connection issues
line3: means the errorinfo_message contains the filter value “Failed to install/upgrade from” will be grouped into connection issues
line9: means the errorinfo contains CouldNotStart will be categorized into start/restart issues.

Architecture overview

Here is the architecture diagram of ARTs.

  1. Jenkins master triggers the master build which triggers slave builds to run the test suites in parallel.
  2. Each slave runs the test suite for specific feature and generates the test result logs.
  3. Each slave generates the test result JUnit xml file as the structured data after the execution.
  4. The Jenkins plugin Splunkins processes the JUnit xml files to formatted JSON data with the metadata and send it to Indexer.
  5. Each slave also forwards the test result log files to Indexer.
  6. Splunk App for Jenkins handles the index and sourcetype ARTs used.

Basic features

Now you can start to use ARTs. Here are several main pages.

Overview page

Overview page will show you the overall information of a set of jobs with the following metrics:

  • job status (running or completed)
  • failed cases/total cases
  • failure delta versus the previous run
  • trend of failures in the last 7 days by default

Clicking on the job will drilldown to the corresponding Trend page.

Trend page

Trend page is a new page introduced in ARTs 2.0. The legacy Summary page in 1.0 lacks of detailed trend info. The new Trend page can be treated as an upgraded version of the legacy Summary page.

It displays trendline for different categories of failures as well as other more detailed job failure information that you can drilldown to view more details.

Overview summary
Title Details
Total tests The total number of all cases (excluded skipped ones)
Pass rate The number of passed cases ÷\div÷ Total tests ×\times× 100%
Total failures The total number of all failed cases
Skipped tests The total number of all skipped cases
Failure category
Title Details
Product bug failures The total number of failures due to product bug
Setup failures The total number of failures due to connection errors/issues in setup step. The connection errors/issues are defined in lookup failure_category.csv (category=connection)
Start/Restart failures The total number of failures due to start/restart errors/issues. the start/restart errors/issues are defined in lookup failure_category.csv (category=start)
Connection failures Similar as setup failures, the only difference is that the error/issue doesn’t occur in setup step.
Flaky failures The total number of failures whose notes are started by flaky, such flaky_*
Test failures The total number of failures due to test script issues.

If you clicking on any of the failure categories, the Triage button will be displayed. Clicking on the button, you will be brought to the Triage page.

Triage page

Triage page is the core page in ARTs. There is also summary trend information stated in triage page.
Failures are distributed by error types or features. You can find all the failures in a specified error type or a specified feature.
Failures are also grouped by “Triaging” and “Untriaged”.

column name description
Triaging being investigated by someone, with triage note started by “_”, such as _investigated_by_xx
Untriaged no triage note, need someone to look into it

Now you can follow the Triage workflow to triage.

Triage workflow

Triage page shows the results of a specified job.

Step1: Select a job you would like to triage.

Select dropdown inputs , such as Product, Source, etc. to narrow down the scope to find the job.

Note:

  • If you can’t find the job name, check the time range. The default time range is “Last 7 days”, the latest build of the job may be out of the range.
  • If there is no result after you set all the inputs, check whether you select the correct Operating System.
  • If you are a feature owner, you may only care about a specified feature. You can select a Feature Group that will only show the results in the feature group.

Step2: Select the build you would like to triage.

There are two builds in the specified job, the right build is always earlier than the left one. (That is different to branch comparison page.)

Note: If there is only one build in the time range (such Last 7 days), the two builds are the same build.

It will show the Overview summary and Failure category for each build.

Note: You can only click on a failure number in the left build to see the details.

Clicking on the Total failures, it will show all the failures in the left build (comparing to the right build).

Column name Definition
J Link directed to Jenkins job
Feature Clicking on it will show all the failures in the specified feature
Failed Total failure number (diff to right one) in one feature
Triaging Failures which are in investigating
Untriaged Failures are not be triaged (very important)
Failure type The failure types occur in the feature

Check the Statistics Charts, it will show you Failures distribution by type and Failures distribution by feature. Clicking on a specified failure type, it will drillldown and show all the failures caused by this failure type. Clicking on a specified feature, it will show all the failures in the feature.

Step3: Triage failures.

Step3.1: Triage Product bug failures.

Clicking on the “Product bug failures” category, it will show the failures summary in the table.
It lists all the failures which caused by a product bug and groups them into features.

Check the checkbox “Show JIRA Tickets”, it will show the details of the corresponding JIRA tickets. The status which are “Resolved” or “Closed” will be highlighted in yellow.

Clicking on the bug_id, it will be directed to the JIRA system. Clicking on other column of the row, it will show the related failure cases. Clicking on the failed case, it will show you the case status trend in Status Matrix Custom Viz and the errorstacktrace. If you find the case still failed with the same error, you may re-open the bug in JIRA system. Otherwise, you need to investigate the new cause.

Note:

  • If you can’t see the bug table, check whether you have enabled the JIRA settings which mentioned in the very beginning of Installation and configuration
  • If the JIRA ticket is not a product bug, such as infrastructure issue INFRA-XXX, you don’t want to categorize it as a product bug. You can append INFRA value to the jira_bug_blacklist in arts_settings.conf
Step3.2: Triage Infrastructure failures.

We defined the infrastructure failures which include Setup failures, Start/Restart failures and Connection failures.

How to define the specified category. Refer to failure_category.csv.

category in failure_category.csv case failed in setup/teardown Failure category
connection yes Setup Failures
connection no Connection failures
start NA Start/Restart failures

You can also change the whole definition. Refer to macros.conf

Step3.3: Triage Flaky failures.

All the cases with triage note which is prefixed with “flaky” may be categorized into Flaky failures.

Note: if you want to update the definition, update the macro flaky_failures in macros.conf.
Refer to Triage note rule to know more about triage note.

Step3.4: Triage Test failures.

This is the most important category since most un-triaged failures are categorized here.

In the case when you find failures in “Untriaged” column of a feature, clicking on the failure number cell will show all the untriaged cases in the feature. (similar to Step3.1 ) You are going to investigate on one of the failures or all the failures that you think are caused by the same issue. You want to inform other team members so that they will not do the same work. Clicking on the Edit Triage Notes button, it will pop out a Triage Dialog.

There are several scopes to add to triage notes.

Scope Definition
current test case Add triage note for current test case on all jobs.
test cases sharing the same test method Add triage note for current test method on all jobs.
current test class Add triage note for current test class on all jobs.
current feature suite Add triage note for current feature suite on all jobs.
test cases with the same error message Add triage note for all cases with same error message
current test case for the specified job name ONLY Add triage note for current test case on only current job.
test cases sharing the same method for the specified job name ONLY Add triage note for current test method on only current job.
current test class for the specified job name ONLY Add triage note for current test class on only current job.
current feature suite for the specified job name ONLY Add triage note for current feature suite on only current job.

Add a triage note like “investigated_by**” and then the failed case will be grouped into “Triaging” column in triage page. If you find the root cause, e.g. a bug with JIRA ticket JIRA-123, you can edit the triage note with “JIRA-123”. Then the failed case will be grouped into Triaged.

Triage note rule
Note Triaging/Triaged failures category
_fixed_by_bob_11_8 Triaging (note started with “_”) test failure
flaky_Test-123 Triaged flaky failure (note started with flaky)
flaky_Bug-123 Triaged product failure
Test-123_found_by_bob Triaged test failure
Bug_123_Test-123 Triaged product_failure (the first JIRA ticket is Bug-123)
Test-123_Bug_123 Triaged test failure (the first JIRA ticket is Test-123)
_Test-123_Bug_123 Triaging (note started with “_”) test failure

The type of Bug-123 is “Bug” in JIRA system
The type of Test-123 is “Test” in JIRA system

Step4: Add a global message. (optional)

If there is a critical issue and you want to inform others, click on button “Add Global Message” on top of the page.

Note: Only the message owner and the user defined in arts_settings.conf (admin_user) can edit/delete other one’s message.

Features in depth

Customize your own overview page

There is a template that helps you create your own overview page. The template is a dashboard named overview_template.xml. You can just clone and update its content.

overview_template.xml

<form hideEdit="True" stylesheet="overview.css, progressview.css"  
  script="render_overview_page_from_xml.js">
  <label>overview template</label>
  <row>  
	 <panel> job_section_1 </panel>
	 <panel> job_section_2 </panel>
  </row>  
  <row> 
     <panel> job_section_3 </panel>
     <panel> job_section_4 </panel>
  </row>
</form>

job_section_i

%data-job-name%: the name of job in jenkins
%data-title%: custom title for the panel
%data-host%: the value of host
%data-os%: windows, linux, mac
%data-threshold%: the panel will be highlight when the failure number exceed threshold, default is 400
%data-earliest-time%: customized timerange for earliest time, such as -7d@h
%data-latest-time%: customized timerange for latest time, such as now

 <html>  
 <div data-job-name="%data-job-name%"
      data-title="%data-title%
      data-host="%data-host%"
      data-os="%data-os%"
      data-threshold="%data-threshold%"
      data-earliest-time="%data-earliest-time%"
      data-latest-time="%data-latest-time%">
  </div>
 </html>

Branch comparison

Similar to the Triage page, there are several inputs you need to select first. But Branch comparison page will help you when you need to compare the test results between different branches.

It also will show the Overview summary and Failure category for a specified run of each job.

Note: You can only click on the failure number in the left build to see the details. (Same as Triage page)

In the Summary Table Fields, you can only select the column name that you are interested in.

Column Name Definition
Feature The feature group
Total Total number (left one comparing to the right one)
Added The number of cases that run in left job build but not run in the right one
Removed Opposite to Added
Fixed The number of cases that pass in the left job build but failed in the right one.
Existing The number of cases failed on both job builds
Regression The number of cases that failed in the left job build but pass in the right one.
New The number of cases that failed which was only running in the left job, such as new added cases.

The Regression and New columns are the ones that you should pay more attention to. Clicking on the number in Regression column, it will show you the regression failures. Clicking on one of the failures, it will show you the status trend of the case running in the right job.

Note: From the status trend, you can find whether the failure is a flaky failure. If yes, it may not be a regression.

You can also follow the steps in the Triage workflow to edit triage notes.

Maintenance

Maintenance page will help you handle:

  • Out-of-date triage notes

It lists those triage notes with all passed test cases, which mean the triage notes are no longer needed. (e.g. the product bug has been fixed). It also lists all the skipped and failed test cases with triage notes as a summary. If you click on any triage note, it will navigate to the tag management page where the delete action is provided.

  • Known bugs / issues

It lists all the failed tests and their corresponding bugs and issues. This page gives you a quick overview of the root causes of the test failures.

  • long running test cases

Some test cases will run a long time without expected. It means that these cases may need to be improved.

FAQ

  1. Q: Why the case is not grouped in Triaging column in triage page after I add triage note to the case?
    A: Only the cases with triage note with prefix “_” will be in Triage column, otherwise it will be grouped into Failed column that means it is triaged.

  2. Q: What’s the rule to category failures, such as product bugs, test failures, etc.?
    A: Here is a diagram help you to understand the rule.

    note: If you won’t to category the failures with triage note such as TNT-123, Auto-123, to product bug failures, you need to add the them to the blacklist in arts_settings.conf. Set the jira_bug_blacklist=TNT|Auto

  3. Q: Does ARTs support other CI system, such as Bamboo?
    A: No.

  4. Q: What kind of test framework does ARTs support?
    A: By default ARTs supports the pytest framework and extracts the errorinfo/errorinfo_message according to the pytest stacktrace. If you find it didn’t meet your requirement, update the marcos in the macros.conf, such as generate_error, extract_errorinfo.

  5. Q: Does ARTs support other Bug system besides JIRA?
    A: No.

  6. Q: In the Branch Comparison page, why one failed case is grouped into New Column but it run in the right job indeed.
    A: You may be confused if you select a specified failure category in Branch Comparison. For example, you select the Test failures category and find one New failure. But it run and also failed in the right job indeed. What happen? After investigating, you find that it failed because of an infrastructure issue. Best practice is select Total failures category.

  7. Q: In the Branch Comparison page, why do not show all the removed cases sometimes?
    A: Branch Comparison pageOnly comparing the features containing cases in the left job build to the right one. There may be jobs with no results. These cases will not be calculated in the Removed column.

  8. Q: Why do I meet the out of memory issue for mvexpand or mvcombine command?
    A: Contact the admin for hardware and memory limits. Admin may change the limits.conf as follows.

     [mvexpand] 
 max_mem_usage_mb=10240
 [mvcombine]
 max_mem_usage_mb=10240
