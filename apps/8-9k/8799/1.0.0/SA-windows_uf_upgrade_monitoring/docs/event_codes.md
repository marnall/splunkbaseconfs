# Splunk Windows Universal Forwarder Upgrade App Event Codes

###################################################

## Overview

This document provides a reference for event codes generated by the Splunk Windows Universal Forwarder Upgrade App solution.

Event codes are used to:

* Track upgrade execution state
* Identify success, warning, skipped, and failure conditions
* Support troubleshooting and operational reporting
* Power Splunk dashboards and automation workflows
* Support retry-loop prevention and upgrade state tracking
* Track intentional retry-control actions
* Track reset actions used to allow future controlled retry waves

The solution is organized into three Splunk apps:

| App                                             | Purpose                                                                    |
| ----------------------------------------------- | -------------------------------------------------------------------------- |
| `SA-windows_uf_upgrade_monitoring`              | Search Head app for dashboards, data model, navigation, and documentation. |
| `TA-windows_uf_upgrade_automation`              | Endpoint-side Windows Universal Forwarder upgrade automation app.          |
| `TA-windows_uf_upgrade_parsing`                 | Parsing and indexing support app for upgrade logs and MSI logs.            |

Event codes are collected from the upgrade application logs and can be queried directly in Splunk using the configured index and sourcetype.

Default index:

```
splunk_upgrade
```

Default primary sourcetype:

```
splunk_upgrade
```

Default MSI log sourcetype:

```
splunk_upgrade_msi
```

## Event Code Ranges

###################################################

| Range       | Component                   | Description                                                                                                                                                       |
| ----------- | --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `9800-9829` | Windows UF Upgrade Workflow | Windows Universal Forwarder upgrade setup, execution, validation, cleanup, retry prevention, and state tracking.                                                  |
| `9830-9839` | Retry Control Workflow      | Retry flag creation, retry marker handling, and retry marker reset events generated by retry-control logic included with `TA-windows_uf_upgrade_automation`. |
| `9840-9899` | Reserved                    | Reserved for future Windows UF Upgrade App operational events.                                                                                                    |

## Severity Definitions

###################################################

| Severity | Description                                                                                                                   |
| -------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `INFO`   | Normal operational event, successful action, skipped condition, expected state update, or intentional retry-control activity. |
| `WARN`   | Potential issue that may require review but does not necessarily indicate failure.                                            |
| `ERROR`  | Failure condition requiring troubleshooting or remediation.                                                                   |

###################################################

| Component                | Description                                                                                           |
| ------------------------ | ----------------------------------------------------------------------------------------------------- |
| Setup / Validation       | Pre-upgrade checks such as MSI discovery, Splunk binary validation, and prerequisite validation.      |
| Version Validation       | Current and target Splunk Universal Forwarder version checks.                                         |

| Retry Control Reset      | Optional disabled-by-default reset logic included in `TA-windows_uf_upgrade_automation`.         |

## Log Sources

###################################################

The Windows upgrade app writes logs to:

```
C:\ProgramData\SplunkUpgrade\
```

The primary log files are:

| Log File                                | Purpose                                                                                                              |
| --------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| `splunk_upgrade_setup.log`              | Setup, validation, retry prevention, state checks, force retry flag consumption, and scheduled task creation events. |
| `splunk_upgrade_exec.log`               | Main upgrade execution, service control, upgrade result, cleanup, and state update events.                           |
| `splunk_upgrade_msi_<timestamp>.log`    | Verbose MSI installer output for each upgrade attempt.                                                               |
| `splunk_upgrade_force_retry_helper.log` | Retry-control and retry-reset operational events.                                                                    |

The JSON state file is written to:

```
C:\ProgramData\SplunkUpgrade\state\splunk_upgrade_state.json
```

The optional force retry flag is:

```
C:\ProgramData\SplunkUpgrade\state\force_retry.flag
```

The force retry helper marker is:

```
C:\ProgramData\SplunkUpgrade\state\force_retry_helper_created.marker
```

## Important Files and Their Purpose

###################################################

| File                                    | Created By                                                     | Consumed / Used By                                     | Purpose                                                                                                      |
| --------------------------------------- | -------------------------------------------------------------- | ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| `splunk_upgrade_state.json`             | Main upgrade workflow                                          | Main upgrade workflow                                  | Tracks installer name, hash, target version, attempt count, status, cleanup state, and `do_not_retry` state. |
| `force_retry.flag`                      | Retry-control logic in `TA-windows_uf_upgrade_automation` | Main upgrade setup script                              | Allows the same installer hash to run one additional time. The setup script removes this file when consumed. |
| `force_retry_helper_created.marker`     | Retry-control logic in `TA-windows_uf_upgrade_automation` | Retry-control and retry-reset logic                    | Prevents retry-control logic from creating multiple retry flags if the retry-control input remains enabled.  |
| `splunk_upgrade_force_retry_helper.log` | Retry-control and retry-reset logic                            | Log monitor in `TA-windows_uf_upgrade_automation` | Captures retry-control and reset activity using event codes `9830-9835`.                                     |

## Retry-Control Workflow Events

###################################################

The retry-control workflow uses a combination of main app state tracking and optional retry-control logic included in `TA-windows_uf_upgrade_automation`.

| Workflow Step                                     | Expected Event Code(s) | Description                                                                                        |
| ------------------------------------------------- | ---------------------: | -------------------------------------------------------------------------------------------------- |
| Upgrade attempt fails                             |                 `9813` | The MSI upgrade failed. Review the `details` field and MSI log.                                    |
| State file written with terminal state            |                 `9821` | The app wrote the state file and should prevent uncontrolled retry loops.                          |
| Same installer is skipped later                   |                 `9819` | The same installer hash was already attempted and `do_not_retry=true`; setup skipped the retry.    |
| Retry-control logic creates retry flag            |                 `9830` | Administrator intentionally enabled the retry-control input to allow one more retry.               |
| Retry-control logic skips because marker exists   |                 `9831` | Retry-control logic already ran once and will not create another flag until the marker is reset.   |
| Main setup script consumes force retry flag       |                 `9820` | The main setup script detected and removed `force_retry.flag`; one retry is allowed.               |
| Retry-reset logic removes marker                  |                 `9833` | Administrator intentionally enabled retry-reset logic so another future retry flag can be created. |
| Retry-reset logic skips because marker is missing |                 `9834` | No marker existed; no reset was needed.                                                            |
| Retry-control or retry-reset action fails         |         `9832`, `9835` | Review the `details` field and endpoint file permissions.                                          |

Important operational rule:

```
Do not intentionally enable the force retry input and retry reset input at the same time during normal operations.
```

Recommended retry sequence:

```
1. Upgrade fails.
2. Main app writes state and prevents uncontrolled retries for the same installer hash.
3. Administrator fixes the underlying issue.
4. Administrator enables the force retry input for selected failed endpoints.
5. Retry-control logic creates force_retry.flag and force_retry_helper_created.marker.
6. Main upgrade setup script consumes force_retry.flag and allows one retry.
7. Administrator disables the force retry input after the intended retry action is complete.
8. If another retry wave is needed later, administrator enables the retry reset input first.
9. Retry-reset logic removes force_retry_helper_created.marker.
10. Administrator disables the retry reset input.
11. Administrator enables the force retry input again when another controlled retry is needed.
```

## Common Fields

###################################################

The app uses key-value formatted logs. Splunk search-time extraction should identify most fields automatically when `KV_MODE = auto` is configured for the `splunk_upgrade` sourcetype.

| Field              | Description                                                                                |
| ------------------ | ------------------------------------------------------------------------------------------ |
| `eventcode`        | Numeric event code representing the operation or condition.                                |
| `status`           | Event status such as `Success`, `Warning`, `Error`, `Skipped`, or similar workflow status. |
| `message`          | Short message describing the event.                                                        |
| `details`          | Additional detail about the event, error, or action taken.                                 |
| `installer_name`   | MSI installer filename.                                                                    |
| `installer_hash`   | SHA256 hash of the MSI installer.                                                          |
| `installer_path`   | Full path to the MSI installer when logged.                                                |
| `target_version`   | Target Splunk Universal Forwarder version extracted from the MSI filename.                 |
| `previous_version` | Splunk Universal Forwarder version before upgrade execution.                               |
| `current_version`  | Splunk Universal Forwarder version after validation or upgrade execution.                  |
| `upgrade_success`  | Boolean-style field indicating upgrade success status when logged.                         |
| `attempt_count`    | Number of attempts tracked for the same installer hash.                                    |
| `last_status`      | Last known state stored in the JSON state file.                                            |
| `cleanup_status`   | Installer cleanup state, such as `cleanup_success` or `cleanup_failed`.                    |
| `do_not_retry`     | State value used to block automatic retries for the same installer.                        |
| `force_retry_file` | Path to `force_retry.flag`.                                                                |
| `marker_file`      | Path to `force_retry_helper_created.marker`.                                               |
| `taskname`         | Windows Scheduled Task name.                                                               |
| `splunkd_status`   | Output from Splunk service status checks.                                                  |
| `msi_log_file`     | Full path to the MSI log generated for an upgrade attempt.                                 |

## Common Troubleshooting Searches

###################################################

### Search for all upgrade app events

```
index=splunk_upgrade sourcetype=splunk_upgrade
| table _time host sourcetype eventcode status message details
| sort -_time
```

### Search for all upgrade failures

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode=9813
| table _time host sourcetype eventcode status message details previous_version current_version msi_log_file
| sort -_time
```

### Search for scheduled task events

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode IN (9802, 9803, 9804, 9805, 9806)
| table _time host sourcetype eventcode status message details taskname
| sort -_time
```

### Search for Splunk service control events

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode IN (9808, 9809, 9810, 9811)
| table _time host sourcetype eventcode status message details splunkd_status
| sort -_time
```

### Search for version validation events

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode IN (9814, 9815, 9816, 9817, 9818)
| table _time host sourcetype eventcode status message details
| sort -_time
```

### Search for retry prevention and state tracking events

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode IN (9819, 9820, 9821, 9822)
| table _time host sourcetype eventcode status message details installer_name target_version installer_hash attempt_count last_status cleanup_status do_not_retry
| sort -_time
```

### Search for force retry events

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode IN (9830, 9831, 9832)
| table _time host sourcetype eventcode status message details force_retry_file marker_file
| sort -_time
```

### Search for retry reset events

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode IN (9833, 9834, 9835)
| table _time host sourcetype eventcode status message details marker_file
| sort -_time
```

### Search for all retry-control activity

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode IN (9820, 9830, 9831, 9832, 9833, 9834, 9835)
| eval retry_control_action=case(
    eventcode="9820", "Force retry flag consumed by setup",
    eventcode="9830", "Force retry flag created",
    eventcode="9831", "Force retry skipped because marker exists",
    eventcode="9832", "Force retry flag creation failed",
    eventcode="9833", "Retry marker reset",
    eventcode="9834", "Retry marker not present",
    eventcode="9835", "Retry marker reset failed",
    true(), "Unknown retry-control action"
)
| table _time host eventcode status retry_control_action message details force_retry_file marker_file
| sort -_time
```

### Search for cleanup events

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode IN (9807, 9823)
| table _time host sourcetype eventcode status message details installer_name
| sort -_time
```

### Search for event code coverage

```
index=splunk_upgrade sourcetype=splunk_upgrade
| stats count dc(host) AS host_count values(host) AS hosts by eventcode status message
| sort eventcode
```

### Search for hosts blocked by retry prevention

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode=9819
| stats latest(_time) AS last_seen latest(status) AS status latest(message) AS message latest(details) AS details latest(installer_name) AS installer_name latest(target_version) AS target_version latest(cleanup_status) AS cleanup_status by host
| eval last_seen=strftime(last_seen, "%Y-%m-%d %H:%M:%S")
| table last_seen host status message installer_name target_version cleanup_status details
| sort host
```

### Search for retry flags created by retry-control logic

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode=9830
| table _time host status message force_retry_file marker_file
| sort -_time
```

### Search for retry marker resets

```
index=splunk_upgrade sourcetype=splunk_upgrade eventcode=9833
| table _time host status message marker_file
| sort -_time
```

## Dashboard Guidance

###################################################

Failure events are retained in Splunk for historical troubleshooting.

A host may generate one or more failure events before later completing a successful upgrade. This is expected behavior.

Recommended dashboard groupings:

| Dashboard Panel                 | Event Codes                                            | Purpose                                                                                     |
| ------------------------------- | ------------------------------------------------------ | ------------------------------------------------------------------------------------------- |
| Setup / Validation Events       | `9800`, `9801`                                         | Shows missing installer and missing Splunk binary issues.                                   |
| Scheduled Task Events           | `9802`, `9803`, `9804`, `9805`, `9806`                 | Shows scheduled task creation, removal, start, and failure events.                          |
| Splunk Service Events           | `9808`, `9809`, `9810`, `9811`                         | Shows Splunkd stop/start success and failure activity.                                      |
| Version Validation Events       | `9814`, `9815`, `9816`, `9817`, `9818`                 | Shows version parsing, skipped upgrades, unsupported upgrade paths, and validation success. |
| Upgrade Result Events           | `9812`, `9813`                                         | Shows final upgrade success or failure.                                                     |
| Cleanup Events                  | `9807`, `9823`                                         | Shows installer removal success and failure.                                                |
| Retry Prevention / State Events | `9819`, `9820`, `9821`, `9822`                         | Shows retry prevention, force retry consumption, and JSON state tracking activity.          |
| Retry Control Events            | `9830`, `9831`, `9832`                                 | Shows retry flag creation, retry-control skips, and retry-control failures.                 |
| Retry Reset Events              | `9833`, `9834`, `9835`                                 | Shows retry marker reset activity.                                                          |
| Retry Control Activity          | `9820`, `9830`, `9831`, `9832`, `9833`, `9834`, `9835` | Shows end-to-end controlled retry operations.                                               |

The primary remediation panel should be:

```
Hosts Not on Target Version
```

Use failure panels for troubleshooting and operational history, not as the sole indicator of current remediation status.

Use retry prevention panels to identify hosts where the MSI remained present but was not retried because the installer had already reached a terminal state.

Use retry-control and retry-reset panels to confirm whether controlled retry actions were enabled and acted as expected.

## Operational Notes

###################################################

* Event codes should remain stable across app versions whenever possible.
* New event codes should be added within the `9800-9899` range unless a new component range is introduced.
* Use consistent field names such as `eventcode`, `status`, `message`, `details`, `host`, and `_time`.
* The app currently logs `eventcode` in lowercase.
* Wrap filenames, paths, sourcetypes, and SPL field names in backticks when documenting them in Markdown.
* A failed installer cleanup event `9807` should not cause repeated upgrade attempts if the JSON state file was written successfully.
* Event code `9819` confirms the retry-loop prevention logic is working.
* Event code `9820` should only appear when an administrator intentionally creates or enables a force retry action and the main setup script consumes `force_retry.flag`.
* Event code `9830` should only appear when retry-control logic creates a new `force_retry.flag`.
* Event code `9831` means retry-control logic did not create a new flag because the marker already exists.
* Event code `9833` means retry-reset logic removed the marker so a future retry flag can be created.
* The force retry input and retry reset input should not be intentionally enabled at the same time during normal operations.
* If the app package has not changed but must be redeployed through Deployment Server, update or add a harmless file such as `force_redeploy.txt` to trigger redeployment.

Example force redeploy marker on Linux/macOS:

```
touch force_redeploy.txt
```

Example force redeploy marker on Windows PowerShell:

```
New-Item -ItemType File -Force .\force_redeploy.txt
```
